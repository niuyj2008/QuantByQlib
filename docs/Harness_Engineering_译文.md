# Harness 工程：为什么 2026 年最优秀的 AI 工程师已经停止亲手写代码

> 原文作者：[@heynavtoor](https://x.com/heynavtoor/status/2037200578842157462)
> 翻译日期：2026-03-28

---

一位研究人员对同一个 AI 模型在同一个编程基准测试上进行了两次测试。第一次得分 42%，第二次得分 78%。同一个模型，同一套测试，其他条件完全相同。

唯一改变的，是 **harness（执行框架）**。

不是提示词，不是模型，不是温度参数。是 harness——包裹在 AI 外层、告诉它如何行动的那套系统：规则、工具、技能文件、记忆文件、反馈循环。

**这是当前 AI 辅助开发领域最重要的发现。** 然而，圈子之外几乎没有人在讨论它。

LangChain 独立验证了同样的结论。他们的编程 Agent 在什么都没有修改的情况下，仅凭改变 harness，就从 Terminal Bench 2.0 排行榜前 30 名之外跃升至前 5 名。OpenAI 也证明了这一点。他们的 Codex 团队构建了一个超过一百万行代码的生产应用，其中没有一行代码由人类亲手编写。工程师们不写代码——他们**设计 harness**。

设计这些 harness 的能力，现在有了专属名称：**Harness 工程（Harness Engineering）**。这个词诞生于数周前，已经在 YouTube 和工程博客上开始流传，但主流媒体几乎没有任何报道。大多数开发者从未听说过这个术语。

这种局面即将改变。率先掌握它的开发者，将获得一种极难被复制的职业竞争优势。

本文是完整指南：Harness 工程是什么，为什么它比你选用哪个模型更重要，今天如何开始实践，以及为什么它是一项零成本却能让你变得不可替代的技能。

## Harness 工程究竟是什么

简单说清楚。

当你使用 Claude Code、Cursor 或 Codex 这类 AI 编程 Agent 时，AI 模型只是整个系统的一部分。模型是大脑，但大脑需要指令、需要工具、需要规则、需要记忆，还需要在错误流入生产之前将其捕获的反馈循环。

**模型周围的这一切，就是 harness。**

可以类比骑马：马本身力量强大，但没有缰绳、马鞍和衔铁，马会随心所欲地跑。Harness 的作用，就是将这股力量引导向你需要的方向。

AI 编程 Agent 也一样。没有精心设计的 harness，它会猜测、会偏离、会反复犯同样的错误，产出的代码看起来令人印象深刻，却在生产环境中崩溃。

**Harness 工程，就是设计和配置这套框架，让你的 AI 编程 Agent 永远不再犯同一个错误。**

Terraform 的创造者、业界最受尊重的工程师之一 Mitchell Hashimoto 这样定义它：

> "每当发现 Agent 犯了一个错误，你就花时间设计一个解决方案，确保 Agent 永远不会再犯这个错误。"

这就是整套哲学，一句话概括。不要祈盼更好的模型——修复模型周围的系统。

## 编程 Agent Harness 的五个调节杆

每个编程 Agent 都有五个可调节的配置点，这就是你的调节杆。拨动正确的杆，同一个模型就能产出戏剧性更好的代码。

### 1. 系统提示词（CLAUDE.md / AGENTS.md）

这是放在你的代码仓库根目录的一个 Markdown 文件，在每次会话开始时被注入 Agent 的上下文。它告诉 Agent 你的代码库是什么、需要遵循什么规范、应该避免什么。

大多数人要么完全跳过这个文件，要么让 AI 自动生成它。两种做法都是错误的。苏黎世联邦理工大学（ETH Zurich）的一项研究测试了来自不同代码库的 138 个 Agent 文件，发现 AI 生成的文件实际上**损害了性能**，同时还多消耗 20% 的 token。人工编写的文件确实有帮助，但前提是简洁且具体。

**规则**：控制在 60 行以内。只包含适用于所有任务的通用指令。不要放目录结构（Agent 自己会发现）。不要放条件规则（"如果做 X，则 Y"会造成混乱）。只放核心内容：技术栈、测试命令、编码规范，以及硬性规定。

### 2. 技能文件（Skills）——知识的渐进式披露

技能是只有当任务匹配时才被加载的指令文件。你不需要把所有东西都塞进系统提示词，而是将知识拆分成聚焦的模块，Agent 按需发现和加载。

例如，你可以为数据库迁移设置一个技能文件，为 API 端点创建设置另一个，为前端组件模式再设置一个。当 Agent 遇到迁移任务时，自动加载迁移技能；其余技能不占用上下文窗口。

这叫**渐进式披露（Progressive Disclosure）**：Agent 以最小上下文启动，随需拉取更多内容。上下文窗口保持干净，Agent 不会因无关指令而混乱。

### 3. MCP 服务器——工具与能力扩展

Model Context Protocol（模型上下文协议）服务器扩展了 Agent 的能力边界，使其不仅限于读取文件和执行命令。它们可以让 Agent 连接 Linear（任务追踪）、Sentry（错误监控）、数据库（实时查询）或任何其他外部系统。

但有一个警告：**每连接一个 MCP 工具，Agent 的系统提示词就会增长一分。** 工具太多会引发 HumanLayer 团队所称的"工具颠抖（tool thrash）"——Agent 把时间浪费在决定用哪个工具上，而不是真正做事。从两三个开始，只有在遇到真实瓶颈时才增加。

### 4. 子 Agent——上下文防火墙

这是最容易被误解的调节杆。子 Agent 并不是要建一个"前端工程师 Agent"和一个"后端工程师 Agent"，那行不通——HumanLayer 团队尝试过，已经放弃了。

子 Agent 的作用是**上下文防火墙**。当主 Agent 遇到一项任务，执行过程中会产生大量中间噪音填满上下文窗口时，它将该任务委托给子 Agent。子 Agent 在独立隔离的上下文中运行，完成工作后只将结果返回，中间步骤不污染主线程。

这就是保持主 Agent 处于"高智模式"的方法。Chroma 的研究显示，AI 模型在更长的上下文中表现可量化地更差。子 Agent 让你将大问题分解成小的、聚焦的会话，让模型始终保持敏锐。

### 5. 钩子（Hooks）——自动化检查点

钩子是在 Agent 工作流的特定节点自动运行的脚本，它们为不确定性系统引入了确定性控制。

例如：pre-commit 钩子可以在 Agent 提交代码前自动运行代码检查工具；pre-completion 钩子可以强制 Agent 在宣布任务完成之前先跑通测试；loop-detection 钩子可以在 Agent 开始反复执行同一个编辑操作时将其捕获。

LangChain 构建了一个叫 `PreCompletionChecklistMiddleware` 的中间件，在 Agent 完成任何任务之前拦截它，强制对照原始需求进行一次验证。就这一个钩子，成为他们整个 harness 中性能提升最大的单项改造。

## 为什么 Harness 比模型更重要

这是大多数开发者理解错误的地方。他们花大量时间争论 Claude vs GPT vs Gemini，追逐每一次新模型发布，相信下一个版本会解决一切。

**数据说的不是这样。**

2026 年 3 月，一项研究证明：仅凭周围的 harness，同一个 AI 模型可以从 42% 的准确率跃升至 78%——几乎翻倍。历史上没有任何一次模型升级带来过 2 倍的性能提升，但一个精心设计的 harness 可以轻松做到。

LangChain 在行业标准基准测试 Terminal Bench 2.0 上验证了同样的结论：同一个模型，不同的 harness，从 30 名开外跳进前 5 名。

OpenAI 的 Codex 团队直接说了：

> "当 Agent 陷入困境，我们把它视为一个信号。我们识别缺少了什么——工具、护栏、文档——然后将其反馈回代码库。"

他们不换模型，他们修 harness。

模型是发动机，harness 是方向盘、刹车和路面。你可以拥有世界上最强大的发动机，但没有方向盘，它只会撞毁。

## 今天就开始 Harness 工程（零成本）

你不需要新工具，不需要付费课程，不需要更换 AI 编程 Agent。你需要改变的，是**遇到失败时的反应方式**。

### 第一步：改变失败反射

- **旧反射**：Agent 犯了错，你手动修复，继续前进。
- **新反射**：Agent 犯了错，你问自己："我怎样确保它永远不再犯这个错误？"然后把解决方案编码进 harness。

这就是整个思维转变。每一次失败都是一个信号，说明你的 harness 缺少某些东西。找出缺少的，补上它，继续前进。Agent 不会再以同样的方式失败。

### 第二步：编写简洁的 CLAUDE.md / AGENTS.md

在代码库根目录创建一个 Markdown 文件，控制在 60 行以内。包含：技术栈、测试命令、硬性规定（"永远不删除迁移文件"、"提交前必须跑通测试"、"使用 TypeScript strict 模式"）。不要有目录结构，不要有条件逻辑，不要有 AI 生成的内容。

### 第三步：构建你的第一个技能文件

识别代码库中一个重复出现的模式：API 端点创建、数据库迁移、组件脚手架……写一个聚焦的指令文件，解释如何正确完成它，包括边界情况和常见错误。将其保存为一个技能文件，Agent 在匹配到该模式时自动加载。

### 第四步：添加一个钩子

从一个 pre-commit 钩子开始，让它在代码检查工具和测试套件上运行。如果 Agent 尝试提交未能通过检查的代码，钩子在其进入代码库之前将其拦截。一个钩子，巨大的影响。

### 第五步：对上下文密集的任务使用子 Agent

当你发现 Agent 在长任务中开始失去连贯性时，将任务拆分成子任务，委托给子 Agent 在隔离环境中完成，只返回结果。主线程保持干净。

### 第六步：每周迭代

每周五，回顾本周发生的失败。针对每一个，向 harness 添加一条规则、一个技能或一个钩子。每次失败五分钟的 harness 工程。随着时间推移，你的 harness 不断积累修复。你的 Agent 每周都变得更可靠——不是因为模型变好了，而是**你的系统变好了**。

## 没有人看到的职业护城河

这就是为什么这件事对你的职业生涯至关重要。

AI 模型正在商品化。每家公司都能访问同样的前沿模型——Claude、GPT、Gemini 对所有人都开放。**模型已不再是竞争优势。**

但一个精心工程化的 harness 是。它是针对你的代码库的，针对你团队模式的，针对你所在领域的边界情况的。它无法通过下载一个模型来复制，它是通过数周乃至数月将真实世界的失败编码进一个不断学习的系统所构建起来的。

能够设计这些 harness 的开发者，是公司无法替代的人。不是因为他们写出了最好的代码，而是因为他们设计了能让 AI 写出最好代码的系统。

OpenAI 明确说过：**工程师的工作不再是写代码，而是设计环境、明确意图、构建反馈循环，让 Agent 能够可靠地工作。**

这就是 Harness 工程。而那些现在就学习它的开发者——在这个术语还很新、这门学科还在成形的时候——将比所有等待观望的人**领先两年**。

提示词工程是 2023 年的技能，上下文工程是 2025 年的技能，**Harness 工程是 2026 年的技能**。

它学习成本为零，无需任何新工具，任何拥有编程 Agent 的开发者都可以立即开始。

唯一的问题是：你是今天就开始工程化你的 harness，还是继续祈盼下一次模型发布来解决一切？

**数据已经给出了答案。**
